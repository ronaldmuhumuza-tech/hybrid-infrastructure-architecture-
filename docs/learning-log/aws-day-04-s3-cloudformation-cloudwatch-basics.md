
The forward slashes create the appearance of folders, but S3 is a flat storage system.

---

## S3 Buckets

Buckets are containers for objects.

Key characteristics:

- Created in a specific AWS region
- Globally unique name (across ALL AWS accounts)
- 3–63 characters
- Lowercase letters and numbers only
- Cannot resemble an IP address
- Default soft limit: 100 buckets per account
- Unlimited objects per bucket

Buckets cannot move between regions once created.

---

## Blast Radius

Because buckets are regional:

- Failure in one region does not affect another region.
- Multi-AZ replication protects against Availability Zone failure.

Cross-region replication must be configured manually.

---

## S3 Use Cases (Patterns)

Good use cases:

- Static website hosting
- Backup storage
- Data lakes
- IoT data ingestion
- Log storage
- Media storage
- Input/output storage for AWS services

Anti-patterns:

- OS boot drives
- High-frequency transactional databases
- Mounting as block storage

# 🏗 AWS CloudFormation Basics

## What is CloudFormation?

CloudFormation is an **Infrastructure as Code (IaC)** service.

It allows you to:

- Create
- Update
- Delete

AWS infrastructure in a consistent, repeatable way using templates.

---

## Templates

Templates are written in:

- YAML (preferred for readability)
- JSON

Both formats achieve the same result.

---

## Template Structure

The only mandatory section is:

- **Resources**

Other common sections:

| Section | Purpose |
|----------|----------|
| AWSTemplateFormatVersion | Template version |
| Description | Explains template purpose |
| Metadata | Controls UI presentation |
| Parameters | User input at deployment |
| Mappings | Static lookup tables |
| Conditions | Conditional resource creation |
| Resources | Defines infrastructure |
| Outputs | Returns useful values |

---

## Logical vs Physical Resources

Inside a template:
              MyWebServer:
              Type: AWS::EC2::Instance


- **Logical Resource** → The name inside the template
- **Physical Resource** → The actual AWS resource created

CloudFormation keeps logical and physical resources in sync.

---

## Stacks

When a template is deployed, it becomes a **Stack**.

Stack lifecycle:

- CREATE
- UPDATE
- DELETE

CloudFormation automatically manages resource dependencies and rollback on failure.

---

## Why CloudFormation Matters

- Enables change management
- Infrastructure version control
- Environment replication (dev, test, prod)
- Disaster recovery automation
- Enterprise-grade automation

# 📊 AWS CloudWatch Basics

CloudWatch is a platform for operational monitoring and observability.

It consists of three core components:

---

## 1️⃣ Metrics

- CPU utilization
- Network traffic
- Disk IOPS
- Service-level metrics

Some metrics are automatically generated by AWS services.  
Additional metrics can be collected using the CloudWatch Agent.

---

## 2️⃣ Logs

CloudWatch Logs ingests:

- Application logs
- System logs
- Custom logs

Used for:

- Troubleshooting
- Security analysis
- Auditing

---

## 3️⃣ Events (EventBridge)

CloudWatch Events (now EventBridge) allows:

- Scheduled automation
- Event-driven responses
- AWS service state change monitoring

Example:
- EC2 instance stops → Trigger Lambda
- Scheduled job every night at midnight

---

# 🔁 Architecture Interaction Model

High-level interaction between services:

User / Application / IoT Device  
        ↓  
S3 (Data Storage)  
        ↓  
CloudWatch (Monitoring & Alerts)  
        ↓  
Automation (SNS, Lambda, etc.)  

CloudFormation deploys and manages all infrastructure components.

- CloudFormation enables Infrastructure as Code.
- Logical resources map to physical resources.
- CloudWatch provides monitoring, logging, and event automation.
- These services work together to form scalable, resilient architectures.

# AWS Day 07.1 – Amazon CloudWatch

**Date: 11/03/2026**  

**Topic:** Amazon CloudWatch Monitoring and Logging

---

# Overview

Amazon CloudWatch is a monitoring and observability service provided by **Amazon Web Services (AWS)** that allows engineers to monitor infrastructure, applications, and services running in the cloud.

CloudWatch collects and processes operational data in the form of:

- Metrics
- Logs
- Events

This data can then be used to:

- Monitor infrastructure health
- Detect anomalies
- Trigger alerts
- Analyse logs
- Automate operational responses

CloudWatch plays a key role in maintaining **reliable, observable, and scalable cloud architectures**.

---

# Core Components of CloudWatch

The main components of CloudWatch include:

- Metrics
- Alarms
- Logs
- Events (via Amazon EventBridge)
- Dashboards

These components together enable engineers to monitor system behaviour and automate responses to operational events.

---

# CloudWatch Metrics

A **metric** is a time-ordered set of data points representing the performance of a resource.

Metrics allow engineers to track the health and performance of AWS services.

Examples include:

- CPU utilisation on **Amazon Elastic Compute Cloud (Amazon EC2)**
- Network throughput
- Disk read/write operations
- Request count on load balancers
- Invocation count for **AWS Lambda**

Each metric contains:

- Namespace
- Metric name
- Timestamp
- Value
- Dimensions

Metrics are collected automatically for many AWS services.

Custom metrics can also be created by applications.

---

# Namespaces

Metrics are organised into **namespaces**.

A namespace groups related metrics together.

Examples include:

AWS/EC2  
AWS/Lambda  
AWS/S3

Custom applications can also create their own namespaces.

Example:

Company/ApplicationName

Namespaces help organise monitoring data logically.

---

# Dimensions

Dimensions are **key-value pairs** used to identify the source of a metric.

Example:

InstanceId = i-1234567890abcdef

Dimensions allow CloudWatch to distinguish metrics generated by different resources.

For example, CPU utilisation can be tracked separately for each EC2 instance.

---

# CloudWatch Alarms

A **CloudWatch Alarm** monitors a metric and performs an action when a threshold is crossed.

Example scenario:

If EC2 CPU utilisation exceeds 80%, an alarm can trigger a notification.

Possible alarm actions include:

- Sending a notification using **Amazon Simple Notification Service (Amazon SNS)**
- Triggering **AWS Lambda**
- Stopping or terminating an EC2 instance
- Executing an Auto Scaling policy

Alarms help automate operational responses and improve system resilience.

---

# Alarm States

CloudWatch alarms operate using three states:

| State | Description |
|------|-------------|
| OK | Metric is within the defined threshold |
| ALARM | Metric has exceeded the defined threshold |
| INSUFFICIENT DATA | Not enough data is available to determine the alarm state |

These states help operators understand the current health of a system.

---

# Amazon CloudWatch Logs

**Amazon CloudWatch Logs** is used to store, monitor, and analyse log data.

Logs may originate from:

- AWS services
- Amazon EC2 instances
- On-premises servers
- Applications
- Hybrid infrastructure environments

CloudWatch Logs is a **regional service**, meaning logs are stored and processed within a specific AWS region.

---

# CloudWatch Logs Architecture

CloudWatch Logs organises log data using the following hierarchy:

Log Group → Log Stream → Log Event

This structure helps organise logs from many sources.

---

# Log Event

A **log event** is the smallest unit of logging data.

Each log event contains:

- Timestamp
- Message

Example log event:

2026-03-10T14:21:33Z  
Failed SSH login from 192.168.1.24

CloudWatch treats the message field as raw log data.

The data may be plain text or structured formats such as **JavaScript Object Notation (JSON)**.

---

# Log Stream

A **log stream** represents a sequence of log events from a single source.

For example, if multiple EC2 instances send operating system logs, each instance creates its own log stream.

This allows logs to remain organised by source.

---

# Log Group

A **log group** is a container for multiple log streams.

Log groups typically represent a specific application or service.

Log groups store configuration settings such as:

- Log retention policies
- Access permissions
- Metric filters

These settings apply to all log streams within the group.

---

# Log Ingestion Methods

Logs can be sent to CloudWatch using several methods.

## Native AWS Integrations

Many AWS services can send logs directly to CloudWatch Logs.

Examples include:

- AWS Lambda
- Amazon Virtual Private Cloud (Amazon VPC) Flow Logs
- AWS CloudTrail
- Amazon Route 53
- Amazon Elastic Container Service (Amazon ECS)

Authentication is handled through **AWS Identity and Access Management (IAM)** roles.

---

## CloudWatch Agent

The **Unified CloudWatch Agent** can collect logs and metrics from:

- Amazon EC2 instances
- On-premises servers
- Hybrid infrastructure

The agent can capture:

- Operating system logs
- Application logs
- Performance metrics

These logs are then transmitted to CloudWatch Logs.

---

## AWS Software Development Kit (SDK)

Applications can send logs directly to CloudWatch using the **AWS Software Development Kit (SDK)**.

This allows developers to integrate logging directly into their applications.

---

# Metric Filters

CloudWatch Logs provides the ability to convert log data into metrics.

This is done using **Metric Filters**.

Metric filters scan logs for specific patterns and generate metrics when those patterns are detected.

Example log entry:

Failed password for root

Metric filter pattern:

Failed password

Each time this pattern appears, a metric value is increased.

This metric can then trigger a CloudWatch alarm.

Metric filters are commonly used for security monitoring.

---

# Example Monitoring Workflow

Application Logs  
↓  
CloudWatch Logs  
↓  
Metric Filters  
↓  
CloudWatch Metrics  
↓  
CloudWatch Alarms  
↓  
Amazon Simple Notification Service (SNS)

This workflow enables automated monitoring and alerting.

# CloudWatch Dashboards

CloudWatch Dashboards provide visual monitoring interfaces.

Dashboards can display:

- Metrics
- Graphs
- Alarm states
- Log insights

Dashboards are commonly used in operations centres to monitor system health in real time.
